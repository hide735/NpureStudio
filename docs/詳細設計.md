# NpureStudio 詳細設計書

## 1. 共通部詳細設計

### 1.1 アーキテクチャ概要
- **モジュール構造**: アプリを以下のモジュールに分割
  - `core/`: WebGPU 初期化、Transformers.js 統合
  - `ui/`: UI コンポーネント（キャンバス、ボタン、入力フォーム）
  - `features/`: 各機能（着せ替え、画像生成）の実装
  - `utils/`: ヘルパー関数（画像処理、ファイル操作）
- **データフロー**: ユーザ入力 → UI コンポーネント → 機能モジュール → WebGPU 処理 → 結果表示

### 1.2 WebGPU 統合
- **初期化**: Safari 26+ で WebGPU を検知し、GPUDevice を取得
- **メモリ管理**: iPhone のメモリ制限（最大 1GB）を考慮し、テクスチャとバッファを効率的に管理
- **エラーハンドリング**: GPU 利用不可時のフォールバック（CPU 処理）

### 1.3 Transformers.js 統合
- **バージョン**: v3 を使用
- **モデル管理**: オンデバイスでモデルをキャッシュ、必要に応じてロード
- **パフォーマンス最適化**: モデルを WebGPU で実行、NPU を活用

### 1.4 共通 UI コンポーネント
- **キャンバス**: HTML5 Canvas または WebGL Canvas を使用、タッチ操作対応
- **ファイルアップロード**: ドラッグ&ドロップとファイル選択をサポート
- **プログレスバー**: AI 処理中の進捗表示
- **エラーメッセージ**: 処理失敗時のユーザーフレンドリーな通知

## 2. 自動着せ替え部詳細設計

### 2.1 機能概要
人物写真と衣服写真を入力とし、AI を用いて仮想試着を実現。

### 2.2 ワークフロー
1. **入力受付**: 人物写真と衣服写真をアップロード
2. **セグメンテーション**: SAM (Segment Anything Model) で人物と衣服のマスク生成
3. **特徴抽出**: IP-Adapter で衣服の特徴を抽出
4. **インペイント**: Stable Diffusion Inpainting で衣服を差し替え
5. **出力**: 合成画像を表示

### 2.3 アルゴリズム詳細
- **SAM モデル**: Transformers.js でロード、人物領域と衣服領域を自動検出
- **IP-Adapter**: 衣服画像から特徴ベクトルを生成
- **Inpainting**: マスク領域を条件付き生成、プロンプトでスタイル調整

### 2.4 UI 設計
- **レイアウト**: 2カラム（左: 人物写真、右: 衣服写真）
- **操作**: タッチでマスク微調整、ピンチでズーム
- **ボタン**: 「試着開始」ボタン、結果プレビュー
- **モバイル最適化**: iPhone 画面サイズにフィット、縦向き優先

## 5. 補足 — `Qwen2-VL-2B-Instruct` の統合案

### 5.1 目的
- マルチモーダルな推論能力を利用して、衣服特徴抽出・マスク改良・インペイント指示の生成・合成結果の検証を高精度に行う。

### 5.2 期待される役割（ワークフローへの割当て）
- Step ④（服の特徴を取得）: `clip-vit-base-patch32` による高速ベクトル抽出を残しつつ、`Qwen2-VL-2B-Instruct` を用いて「自然言語による詳細記述」や「属性（質感・細部パターン・部分的な縫い目等）の階層化ベクトル」を生成。これによりインペイントやマッチングの条件が豊かになる。
- Step ⑤（形の再現（ワープ））: ワープのための制御点や重みの初期推定を `Qwen2-VL` が生成（例: キーランドマーク間の対応関係、優先的に保つ領域）し、Thin Plate Spline 等の数学的処理に渡す。
- Step ⑦（体の隙間の補正／インペイント）: インペイント用の「詳細プロンプト」「局所ストロークガイド」「候補パッチのスコア評価」を `Qwen2-VL` に生成させ、軽量な on-device inpainting に与えることで描き直し品質を向上させる。合成後は `Qwen2-VL` による整合性チェック（色・ライティング・枝分かれ不整合の検出）を行う。

### 5.3 実行モードと現実的な選択肢
- ブラウザ（オンデバイス）実行: `onnx-community/Qwen2-VL-2B-Instruct` を ONNX 化して WebGPU（WebNN / WebGPU via WASM）で実行する選択。量子化（4/8-bit）とレイヤードロップ等の最適化が必要で、メモリ/速度は端末次第（iPhone の上位モデルで実用的）。
- ハイブリッド（推奨初期案）: 軽量処理は完全にブラウザで行い、`Qwen2-VL` はサーバ（ONNX Runtime / GPU）で動かす。モバイルでのレイテンシを許容できる場合のみオンデバイス版を検討。
- フォールバック: `Qwen2-VL` が利用できない場合は、`clip-vit-base-patch32` とルールベース後処理で代替する（機能は限定的になるが軽量・確実）。

### 5.4 API 化の提案（モジュール境界）
- `qwen_vl.generateDescriptors(image, mask, options) -> {textDescription, attributes, embedding}`
- `qwen_vl.suggestWarpControls(sourceImage, targetPose) -> {controlPoints, weights}`
- `qwen_vl.inpaintGuidance(image, mask, context) -> {prompt, strokeHints, candidatePatches}`
- `qwen_vl.verifyComposite(compositeImage, referenceImage) -> {score, issues[]}`

### 5.5 実装上の注意点
- モデルサイズとメモリ: 2Bモデルでも量子化が必須。ブラウザ向けにはモデル分割・ストリーミングやオンデマンドロードを組み合わせる。
- レイテンシ制御: ユーザ操作感を維持するため、重い処理は非同期キューに入れ、サーバ処理の結果で最終調整する戦略が現実的。
- プライバシー: 画像をサーバ転送する場合は、ユーザ説明と許諾を必須にする（オンデバイス実行を優先する方針をドキュメントに明記）。

### 5.6 改訂ワークフロー（要約）
1. 元画像の服をマスク（SegFormer 等）
2. 元画像の背景をマスク（MODNet 等）
3. 服画像から服以外をマスク（SAM）
4. 特徴取得: `clip` による高速埋め込み + `Qwen2-VL` による詳細記述／属性抽出
5. 服の形を再現（Thin Plate Spline 等）: `Qwen2-VL` が制御点候補を提案
6. 重ね合わせ合成（Canvas/WebGL）
7. 体の隙間補正（インペイント）: `Qwen2-VL` が高精度ガイダンスを生成し、軽量inpaintingで描き直す。最終検証も `Qwen2-VL` が担当

---
注: 上記は `Qwen2-VL-2B-Instruct` を「補助的な賢い指示系」および「マルチモーダル検証器」として位置付ける案です。ブラウザ単独での完全オンデバイス実行は最適化次第で可能ですが、まずはハイブリッド運用で開発し、段階的にオンデバイス化を目指すことを推奨します。

## 6. 実装要素の具体化：SegFormer / SAM（WebGPU）・Qwen2-VL（ONNX）・OpenCV.js（WebGPU）

### 6.1 要望の位置付け
ユーザ要望（例：「襟元をもう少し左に寄せて」）を同一ブラウザ内で解釈し、数値（制御点のオフセット、重み、ストローク指示など）に変換して即時に変形・合成に渡す流れを実現する。

### 6.2 セグメンテーション（高速マスク生成）
- 対象: `SegFormer`（人物・服部位）と `SAM`（服切り出し）のWebGPU実装を想定。
- 実行手段: Transformers.js または ONNX → WebNN / WebGPU 実行バックエンドでの量子化モデル（4/8-bit）を用いる。
- 出力: 部位別マスク（例: 上着、パンツ、襟、袖）およびラベル付きマスク（confidence score を付与）。
- 性能目安: モデル量子化と小モデル(b0相当)で、入力解像度を抑えれば「数十〜数百ミリ秒」、強力端末では数ミリ秒クラスの応答に近づく。

### 6.3 Vision LLM（`Qwen2-VL` ONNX）による指示の数値化
- 役割: 自然言語指示を受け、画像領域に対応する数値化された操作（例: 制御点 delta、優先領域スコア、局所色補正パラメータ、インペイントストローク）を出力する。
- 出力仕様（例）:
  - `warpControls`: [{x,y,dx,dy,weight}] — TPS/warp に使う制御点と推奨オフセット
  - `inpaintHints`: [{x,y,radius,strokeVector,confidence}] — インペイントで使う局所ストローク
  - `attributes`: {texture: 'silky', gloss:0.2, colorHist:[...]} — 合成評価や色補正用
  - `naturalLanguageSummary`: "Move collar 10px left and preserve seam details"
- 実行方法: `onnx-community/Qwen2-VL-2B-Instruct` を ONNX 形式で用意し、推論はまずサーバ版（ONNX Runtime/GPU）で検証。オンデバイスは量子化・分割・レイヤードロップで段階的に移行。
- 入出力の設計: 画像は縮小した埋め込み（CLIP 互換）＋高解像度座標マップ（control grid）を合わせて入力し、上記 JSON 構造を返すAPIを定義する。

### 6.4 変形・合成（OpenCV.js + WebGPU）
- 変形処理: Thin Plate Spline（TPS）または移動最小二乗（MLS）を WebGPU 加速で実行。`OpenCV.js`（WASM）に WebGPU 加速レイヤを組み合わせるか、専用のWebGPUシェーダで高速化。
- 指示受理: `Qwen2-VL` の `warpControls` を直接マップして制御点に適用。値はピクセルオフセットか正規化座標で受け渡す。
- 合成: Canvas 2D / WebGL レイヤにマスクを適用して重ねる。必要に応じて局所的な色補正（ガンマ・トーンマッピング）を `attributes` で補正。

### 6.5 体の隙間補正（インペイント）とパイプライン統合
- ワークフロー: マスク生成 → `Qwen2-VL` 指示（warpControls + inpaintHints）→ OpenCV.js で変形→ 軽量 inpainting（オンデバイス）で局所リタッチ→ `Qwen2-VL` による整合性スコア
- on-device inpainting 候補: 小型のDiffusionベース inpaint を検討、または学習済みPatchGAN風の軽量ネットで差分補正。
- 確度向上: `Qwen2-VL` の confidence を閾値にして、低い場合はサーバ側精細処理にフォールバック。

### 6.6 入出力インターフェース（例）
- `seg.run(image) -> {masks: [{label, maskImage,score}]}`
- `qwen_vl.infer(image, masks, prompt) -> {warpControls, inpaintHints, attributes, summary}`
- `warp.apply(sourceImage, warpControls) -> warpedImage` (OpenCV.js)
- `inpaint.run(image, mask, hints) -> repairedImage`

### 6.7 実装上の現実的制約と対策
- モデルサイズ: `Qwen2-VL-2B` は量子化しても容量が必要。まずはサーバ連携でプロトタイプを作る。
- レイテンシ: 重要なUI操作はクライアント小処理で完結させ、重い最終調整は非同期で行う。
- 精度と UX のバランス: LLM の出力は確率的でブレるため、`warpControls` はソフト制約（重み）で扱い、ユーザが微調整できるUIを用意する。

## 7. 軽量インペイント候補と Qwen2-VL ガイダンス方式（詳細）

### 7.1 目的
- モバイル（ブラウザ）上での「局所的で自然な描き直し」を低レイテンシかつ小メモリで実現するための候補設計と、`Qwen2-VL` を用いたガイダンス統合方式を提示する。

### 7.2 候補アプローチ（優先度順）
1) ハイブリッド古典→軽量NN（推奨）
  - まず `PatchMatch` / `telea` 等の OpenCV inpaint を高速パスとして適用し、洞や大きな欠損を埋める。
  - 続けて小型NN（U-Net系の小型版、depthwise-separable conv / gated conv採用）でディテール補正。
  - 利点: 安定性と低メモリ、プログレッシブな品質向上。

2) 小型学習ベース Inpainting（オンデバイス用に設計）
  - アーキテクチャ例: shallow U-Net (3-5倍畳み込みブロック) + GatedConv/PartialConv を簡易化した実装。
  - 容量目安: FP32で20–80MB、量子化8bitで半分、4bitでさらに縮小可能。
  - 実装ソース候補: PartialConv / GatedConv の公開実装をベースに蒸留・剪定して ONNX 出力。

3) スパースPatchGAN/差分修正ネット（超軽量）
  - 入力は「元画像」「候補補正差分」「局所マスク」「ガイダンスストローク」で、出力は局所差分（残差）だけを生成する設計。
  - 利点: 生成負荷が低く高速。細部補正用途に向く。

4) フルDiffusionは非推奨（オンデバイスでは重すぎる）

### 7.3 実現手段（技術スタック）
- モデル実行: `ONNX Runtime Web` (WASM / WebGPU)、または WebNN 実装。WebGPU バックエンドが使える環境を優先。
- 推論最適化: 量子化（8→4bit）、静的パラメータ圧縮、レイヤードロップ、重み共有。ONNX の `quantize_dynamic` → ONNX Runtime で検証。
- 古典手法: `OpenCV.js`（WASM）を最初の高速パスに利用。

### 7.4 Qwen2-VL によるガイダンスの具体例
- 入力: 縮小画像（CLIP互換の埋め込み） + 高解像度座標（制御グリッド） + マスク + ユーザ指示（自然言語）
- 出力フォーマット（JSON 例）:
```
{
  "warpControls": [{"x":120,"y":200,"dx":-10,"dy":0,"weight":0.85}],
  "inpaintHints": [{"x":130,"y":210,"radius":18,"strokeVector":[0.2,-0.8],"confidence":0.9}],
  "attributes": {"colorShift":[-5,3,0],"texture":"matte","preserveEdges":true},
  "summary":"Move collar 10px left; keep seam detail"
}
```
- 利用法:
  - `warpControls` を OpenCV.js の TPS/MLS へ渡して事前変形を実行。
  - `inpaintHints` を「ストロークマップ」や「guided edge map」として NN に入力（入力チャネル追加）。
  - `attributes` は色補正（色ヒストグラムのオフセット）やブレンディング係数に利用。

### 7.5 ブラウザ組合せワークフロー（簡易オーケストレーション）
1. 高速マスク: `SegFormer/SAM` を呼び、服/体マスクを得る。
2. 低レイテンシ補正: `OpenCV.js`（PatchMatch/telea）で穴埋め → すばやく表示。
3. 高品質補正: `Qwen2-VL` に指示を与え、`warpControls`/`inpaintHints` を取得。
4. 変形: `warp.apply()` で TPS を実行（OpenCV.js or WebGPU shader）。
5. 局所NN補正: 小型 ONNX モデルにマスク＋ストロークを与え、差分を生成。
6. 最終合成: color/lighting を `attributes` で調整 → ユーザ確認 → 必要ならサーバで精細処理。

### 7.6 実装のコードスニペット（概念）
```
// 概念的なオーケストレーション
async function refineWithQwen(image, masks, userPrompt) {
  const qout = await qwen_vl.infer(image, masks, userPrompt);
  const warped = await warp.apply(image, qout.warpControls);
  const fast = await openCV.inpaintFast(warped, masks.bodyMask);
  const refined = await onnxInpaint.run(fast, masks.holeMask, qout.inpaintHints);
  const final = colorAdjust(refined, qout.attributes);
  return final;
}
```

### 7.7 評価指標と検証プロセス
- 速度: 目標 50–300ms（小処理）、重処理は非同期で 1–3s。端末別ベンチを必須化。
- 品質: LPIPS / FID（プロトタイプ比較）＋ユーザ評価（主観スコア）。
- フォールバック運用: `Qwen2-VL` confidence が低い場合はサーバ精細処理へフォールバック。
